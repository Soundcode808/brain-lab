## 0. 한 줄 요약 (베스퍼님용)

**"더 똑똑한 프롬프트"가 아니라 "잘 설계된 루프"가 핵심이다.**
AI에게 매번 한 단계씩 시키는 게 아니라, **① 만들고 → ② 점수 매기고 → ③ 좋은 것만 남기고 → ④ 살짝 바꿔서 → ①로 돌아가기**를 자동으로 반복시키면, AI가 **스스로 더 나아진다.**

이때 **가장 중요한 건 "②점수를 어떻게 매기느냐"(=신호)**다. 점수가 정직하면 루프가 알아서 일하고, 점수가 허술하면 AI가 점수만 따먹는 꼼수(reward hacking)를 부린다. 이게 이 조사 전체의 핵심 교훈이다.

---

## 1. 모든 자기개선 루프의 공통 뼈대

세상의 모든 자기개선 시스템은 결국 **같은 루프**다. 부품만 바꿔 끼운다.

```
   변형 생성  →  평가(점수)  →  선택/보관(다양성 유지)  →  변이  →  (반복)
   generate      evaluate       select/archive            mutate
```

설계에서 **딱 하나만 고른다면 → "평가 신호(fitness signal)"**. 검색·개선은 결국 "무엇을 향해 최적화하는가"만큼만 좋아진다.

## 2. 핵심 패턴 5가지 (신뢰도 순)

| 패턴 | 작동 방식 | 피드백 신호 | 신뢰도 |
|------|----------|------------|--------|
| **ReAct** (기본 루프) | 생각→행동(도구)→관찰 반복 | 환경의 실제 관찰 | 모든 루프의 바탕 |
| **Self-Refine** | 한 모델이 생성→자기비판→수정 | 말로 하는 자기 피드백 | 글·스타일엔 OK, **추론엔 위험** |
| **Reflexion** | 실패를 말로 반성 → 기억에 저장 → 다음 시도에 활용 | 성공/실패 결과 + 기억 | 여러 번 재시도 과제에 강함 |
| **Evaluator–Optimizer** | 생성자 + **별도의** 심판이 비판 → 수정 | 분리된 평가자의 비판 | 기준 명확할 때 강함 |
| **Test-driven (테스트 기반)** | 테스트 통과/실패가 루프를 굴림 | **객관적·자동·사람 불필요** | ⭐ **가장 신뢰도 높음** |

**가장 중요한 통찰:** 테스트 기반 루프가 제일 믿을 만하다. "통과 아니면 실패" — AI가 **둘러댈 여지가 없는** 신호이기 때문. 반대로 **한 모델이 자기를 채점하는 자기비판은 가장 약하다**(특히 추론에서 오히려 틀린 답을 확신하게 만듦 — Huang et al. 2024, ICLR).
→ 실전 해법: **생성자와 심판을 분리하라.** "AI는 자기 작업을 후하게 채점한다."

### 2026 신상 변종
- **Self-Rewarding LM** (Meta/NYU): 모델이 스스로 채점 → 그 점수로 자기 가중치를 학습. 3회 반복 후 GPT-4-0613 등을 앞섰다고 보고. *출력이 아니라 두뇌 자체*를 개선.
- **Darwin Gödel Machine** (Sakana, 2025): 에이전트가 **자기 코드를 직접 고치고** 벤치마크로 검증 → 좋으면 보관(아카이브). ~80회 반복 만에 자기 도구를 스스로 개선. "더 나은 에이전트를 만드는 AI."
- **AlphaEvolve** (DeepMind, 2025): LLM이 코드를 진화시켜 **56년 만에 Strassen 행렬곱(49→48회) 개선**, 구글 데이터센터 연산 0.7% 회수 등 *실제 성과*. (4절 참고)
- **Ralph Wiggum 루프** (Geoffrey Huntley, 2026): 한 번에 한 과제씩, **매번 깨끗한 컨텍스트**로 에이전트를 다시 돌리는 단순 bash 루프. **진척은 컨텍스트가 아니라 파일·git에 쌓인다.** → "2026은 Ralph 루프의 해."

## 3. 자기개선 루프의 5대 설계 재료

외우기: **신호 · 기억 · 연산자 · 정지 · 가드레일**

1. **신호(Signal)** — "개선됐는지 어떻게 아나?"
   신뢰도 순: ① **객관적/결정적**(테스트·벤치마크·실측치, 못 속임) > ② 별도 심판 > ③ 학습된 보상모델 > ④ 자기채점(최약).
   격언: *"측정할 수 있으면, 그에 맞춘 개선은 자동화할 수 있다."* → **평가(eval)가 진짜 해자(moat).**

2. **기억(Memory)** — 반복 사이에 무엇이 넘어가나
   컨텍스트 안에서만 / 반성 버퍼(Reflexion) / **파일·git**(Ralph) / **아카이브·집단**(AlphaEvolve·DGM — 다양한 과거 해를 보관해 한 우물에 빠지지 않음).

3. **개선 연산자(Operator)** — 새 후보를 *어떻게* 만드나
   말로 비판→수정 / **LLM이 코드를 변이**(FunSearch·AlphaEvolve) / 자기 코드 재작성(DGM). ※ 가장 값진 수정은 종종 후보가 아니라 **루프 자체**를 고치는 것.

4. **정지 조건(Stop)** — 다음 중 먼저 오는 것에서 멈춤
   검증된 "완료"(테스트 통과·임계치 도달) / **반복 횟수 상한** / **비용·토큰 예산** / **진척 없음 감지** / 회로차단기.
   비용 예시: 20회 루프 ≈ $3.3, 무한 500회 ≈ $80+.

5. **가드레일(Guardrails)** — 신호 따먹기·붕괴·폭주 방지
   **#1 실패: 보상 해킹** — 생성자와 평가자가 같은 모델이면 점수만 오르고 *진짜 품질은 떨어진다*(arXiv:2407.04549). 해법: 평가자 분리 + 객관 신호 + 사람 감독 + held-out(숨긴) 평가.

| **FunSearch** (DeepMind, Nature 2024) | LLM이 함수 생성→자동 채점→좋은 것 보관→반복 | LLM 최초의 *수학적 발견*(cap set 문제 개선) | 진짜 평가자 + 사람이 읽을 수 있는 짧은 프로그램 진화 |

| **TDD 코딩 에이전트** | 실패 테스트→최소 코드→실행→오류 읽고 수정→초록불 | flexbox 엔진 ~800줄+테스트 350개를 3시간에 | 객관적 테스트 신호(못 속임) |

## 5. 왜 "지금" 루프가 화제인가 (흐름)

**프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → 하네스/루프 엔지니어링**

- **1→2단계:** Karpathy가 "컨텍스트 엔지니어링" 유행시킴(컨텍스트 창에 *딱 맞는 정보*를 채우는 기술). "프롬프트는 가고 컨텍스트가 왔다."
- **2→3단계:** *"프롬프트 그만 쓰고 루프/하네스를 설계하라."*
  - **하네스** = 모델을 감싼 모든 것(도구·컨텍스트 정책·훅·샌드박스·서브에이전트·**피드백 루프**).
  - 핵심 명제: **"좋은 하네스를 가진 평범한 모델이, 나쁜 하네스를 가진 뛰어난 모델을 이긴다."**
  - "통제 단위가 프롬프트 → 세션/컨텍스트 → 시스템/하네스로 올라갔다."
  - 출처: OpenAI "Harness engineering", Anthropic "Effective harnesses for long-running agents", Addy Osmani 등.
- **"평가가 해자" 흐름:** "측정할 수 있게 되는 순간, 24시간 돌아가는 자기개선 루프가 곧 경쟁우위가 된다."

→ **요컨대 베스퍼님이 들으신 "이제 단계가 아니라 루프" 가 바로 이 3단계 전환이다.**

## 6. 거품 vs 진짜 (결론)

- ✅ **확실히 통하는 것:** **객관적·외부적·속이기 어려운 신호**를 가진 루프(테스트·벤치마크·실측). + 열린 탐색이 필요하면 **다양성 아카이브**.
- ⚠️ **약한 것:** 외부 신호 없는 **순수 자기비판**(추론을 오히려 악화시킴). → 심판을 분리하라.
- 🔴 **1순위 실패:** 보상 해킹(생성자=평가자일 때).
- 🔴 **2순위 실패:** 정지 조건 없음 → 비용 폭주.
- 🎈 **가장 과대광고, 가장 미검증:** 완전 자율 "AI 과학자"(끝까지 자동 연구) — 방향은 흥미롭지만 독립 평가에선 품질 들쭉날쭉.

**프로젝트가 기억할 한 문장:**
> 자기개선은 더 똑똑한 프롬프트가 아니다. **믿을 수 있는 신호 + 교훈을 나르는 기억 + 변형을 만드는 연산자 + 정지 조건 + 신호 따먹기 방지 가드레일** = 루프. 신호만 제대로 잡으면(객관 > 분리심판 > 자기채점) 루프가 알아서 일한다.
